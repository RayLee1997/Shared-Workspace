# GEMINI.md

## 1. 知识图谱与上下文 (Knowledge Graph & Context)

### 工作区目标 (Workspace Purpose)

本核心工作区是 Ray 的**第二大脑与投资研究指挥中心**。其核心目标是：

1. **深度投资研究**：生产达到机构级水准的个股深度分析报告、行业研究与宏观策略。
2. **技术与知识沉淀**：系统化整理技术栈（如 AI、全栈开发）、生产力工具与方法论。
3. **长期价值积累**：通过结构化的知识管理，将碎片化信息转化为可复用的智慧资产。

### 分类体系 (Taxonomy)

文件夹结构遵循 **P.A.R.A.** (Projects, Areas, Resources, Archives) 与 **JD 体系** (Johnny Decimal) 的混合变体，强调行动导向与领域深度的平衡：

* **00_Inbox (收集箱)**: 稍后读、临时笔记、未分类的原始素材。
* **01_Research (研究)**: 通用课题研究、学术论文笔记、待消化的深度内容。
* **02_Ideas (灵感)**: 创新点子、写作灵感、随机的想法碎片。
* **03_Productions (产出)**: 已发布的文章、视频脚本、公开项目文档 (如 MarkdownToStat, SimpleRSS)。
* **04_Investments (投资)**: **[核心区域]** 个股深度报告（如 TSM, NVDA）、财报分析、投资策略、观察列表。
* **05_Technology (技术)**: 编程语言、框架、AI 模型、技术教程与笔记。
* **06_Accounts (账户)**: 个人事务、财务记录、订阅管理。
* **07_Investigation (调查)**: 专项调查报告、深度尽职调查与背景研究 (如 AI 泡沫, China Export)。
* **08_Personal (个人)**: 个人日记、生活记录、隐私信息。
* **09_MOC (地图)**: Map of Content，知识索引页与导航大纲。
* **11_Explore (探索)**: 新领域探索、实验性项目、前沿趋势追踪 (如 Person, Study)。
* **99_Archives (归档)**: 已完成的项目、过时的资料、历史快照。

*以下文件夹不作为主要分类，但为系统重要组成部分：*

* **gemini-scribe**: Agent 系统配置、会话日志与 Prompt 管理。

## 2. 内容标准与风格指南 (Content Standards)

### 语调与声音 (Tone & Voice)

* **专业投研 (04_Investments)**: 极度理性、客观、数据驱动。类似顶级投行（Goldman Sachs, Morgan Stanley）的研报风格。强调逻辑严密性与数据的准确性。
* **技术笔记 (05_Technology)**: 清晰、简洁、结构化。强调 "How-to" 与原理机制的解释，多用代码块与流程图。
* **调查报告 (07_Investigation)**: 深度挖掘，证据链完整，注重事实核查与多源验证。
* **通用写作**: 具有洞察力，避免废话，注重信噪比。

### 语言规范 (Language Rules)

遵循 **双语原则 (Bilingualism)**，精准桥接宏观战略与微观执行：

* **中文 (Chinese)**: 用于综合叙述、逻辑推演、地缘政治分析、投资策略总结、复杂的定性判断。
* **英文 (English)**: 严格用于专业术语、技术栈名称、财务指标 (Revenue, YoY, CAGR, FCF)、代码实现、API 文档引用。**除非有公认且无歧义的中文译名，否则优先保留英文原文。**

### Markdown 规范 (Markdown Rules)

* **标题层级**: H1 (`#`) 仅用于文件标题；正文从 H2 (`##`) 开始；H3/H4 用于子章节。
* **链接风格**: 优先使用标准 Markdown 链接 `[文本](链接)` 以保持通用性，特定知识关联可使用 Obsidian 双链 `[[链接]]`。
* **元数据 (Frontmatter)**: 关键文档（尤其是研报）必须包含 YAML Frontmatter，记录 `date`, `tags`, `status` 等。
* **强调**: 使用 **加粗** 标记核心观点与关键数据，让读者扫描即可获取要点。
* **引用**: 使用 `>` 块引用来展示摘要、免责声明或关键洞察 (Key Insights)。
* **标注 (Callouts)**: 善用 Obsidian Callouts (如 `[!IMPORTANT]`, `[!NOTE]`) 突出关键信息。

### 数据可视化 (Diagramming)

* **工具**: 强制使用 **Mermaid.js**。
* **风格**: 遵循 "Healing Dream" (治愈梦幻风) 配色系统，保持视觉上的专业与优雅。
* **类型**:
  * **股价/财务趋势**: 使用 `xychart-beta` (折线图/柱状图)。
  * **营收结构**: 使用 `pie` (饼图)。
  * **流程/架构**: 使用 `flowchart` 或 `sequenceDiagram`。
  * **复杂关系**: 使用 `sankey-beta` (桑基图) 展示资金流向。

## 3. 思维模型与方法论 (Mental Models & Methodology)

### 核心思维模型 (Thinking Frameworks)

Agent 在协助时应调用以下模型：

1. **第一性原理 (First Principles)**: 剥离表象，回归事物最基本的真理（如分析芯片股时，回归到制程物理极限与摩尔定律）。
2. **MECE 原则**: 相互独立，完全穷尽。在分析行业竞争格局或风险因素时，确保无遗漏且无重叠。
3. **二阶思维 (Second-Order Thinking)**: 不止看直接后果，更要看后果的后果（如投资分析中，不仅看财报当期表现，更要推演 CapEx 激增对未来 3 年 FCF 的影响）。
4. **金字塔原理**: 结论先行，以上统下。

### 审查标准 (Review Criteria)

一份“优秀”的文档必须具备：

* **数据支撑**: 任何定性判断（"增长强劲"）后必须跟随定量数据（"+35.9% YoY"）。
* **视觉增强**: 复杂的数字关系必须转化为图表。
* **行动导向**: 结尾必须有明确的结论（如 "Strong Buy"）或下一步行动计划。

## 4. Agent 动作指令 (Agent Actions)

### 起草 (Drafting)

* **研报**: 必须先建立大纲，包含“执行摘要”、“公司概览”、“财务分析”、“估值”、“风险”、“做多/做空逻辑”。
* **技术文档**: 必须包含背景、环境配置、核心代码/步骤、常见问题。

### 润色 (Refining)

* **检查**: 自动修正中英文混排格式（在中文与英文/数字间增加空格）。
* **精简**: 删除冗余的修饰词，确保每一句话都有信息增量。
* **排版**: 统一修正 Mermaid 图表配色，确保符合审美标准。

### 归档 (Archiving)

* 当一个项目完结或信息失效时，将其移动至 `99_Archives`，并在原位置留下索引（如果需要）。
